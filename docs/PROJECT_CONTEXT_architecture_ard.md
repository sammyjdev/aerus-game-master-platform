### Turn Data Flow

```text
Browser (React)
  -> WebSocket message with a player action
main.py / connection manager
  -> validates JWT and identifies player_id
game_master.wait_for_batch()
  -> collects simultaneous actions within the batch window
game_master.process_turn()
  1. load campaign state from SQLite
  2. retrieve relevant lore from ChromaDB
  3. assemble the multi-layer prompt context
  4. select the LLM client (BYOK or fallback)
  5. request a streamed model response
  6. parse structured GM output
  7. apply state changes atomically in SQLite
  8. update memory asynchronously
  9. emit gameplay events (death, level-up, mutation, loot, etc.)
connection manager
  -> broadcasts narrative tokens and structured events to all connected clients
```

## 4. ARD - Application Requirements Document

### Functional Requirements

#### FR-01: Authentication and Session
- Single-use invite code generated by an admin.
- JWT session stored on the client for the MVP.
- Silent token refresh before expiration.
- Planned migration to HttpOnly cookies after domain setup.

#### FR-02: BYOK (Bring Your Own Key)
- Players can register their own OpenRouter key.
- Keys are encrypted before persistence.
- Admin covers shared overhead and fallback usage.
- If a player has no valid key or credits, the system can fall back to the admin path.

#### FR-03: Isekai Framing
- Players arrive in Aerus as summoned outsiders.
- Each character chooses a faction during creation.
- Each faction has a public agenda and a secret objective.
- The GM reveals inter-player conflict progressively through narrative memory.

#### FR-04: Character Creation
- Player defines name, race, faction, and a freeform backstory.
- The initial class is inferred from the backstory.
- Base attributes start at 10: STR, DEX, INT, VIT, LUK, CAR.
- Initial inventory should reflect the character's narrative background.

#### FR-05: Game Interface
- Narrative streams token by token.
- Dice events can pause the stream and trigger UI feedback.
- Paused campaign state supports a distinct visual scene.
- Dead players become spectators and can create a new character.
- Audio and image cues are generated asynchronously.

#### FR-06: Combat
- Freeform actions outside combat.
- Initiative based on DEX in combat.
- Natural 20 is a critical success; natural 1 is a critical failure.
- Enemies surface with explicit combat properties when needed.
- Turn batching supports simultaneous party actions.

#### FR-07: Progression
- XP is granted for combat, diplomacy, and major objectives.
- Campaign cap: 500 total attribute points.
- Per-attribute cap: 250.
- Maximum level: 100.
- Class mutations happen automatically at progression milestones.
- Passive milestones unlock every 25 points invested in an attribute.

#### FR-08: Permadeath
- Reaching 0 HP moves the character into spectator mode.
- Narrative history remains preserved.
- A new sheet can be created without leaving the session.

#### FR-09: Image Generation
- Bosses and character creation can trigger asynchronous image generation.
- Images are stored and pushed to clients via events.

#### FR-10: HTTP Endpoints

### Non-Functional Requirements

| ID | Requirement |
| --- | --- |
| NFR-01.1 | End-to-end latency under 5 seconds, excluding image/music generation |
| NFR-01.2 | Prompt context hard cap at 4,000 tokens |
| NFR-02.1 | SQLite must run in WAL mode |
| NFR-02.2 | Every write must be atomic |
| NFR-02.3 | WebSocket reconnection uses exponential backoff |
| NFR-03.1 | BYOK keys stay encrypted and env-backed |
| NFR-03.2 | Long-lived JWT with silent refresh support |
| NFR-03.3 | Explicit CORS allowlist, never wildcard in production |
| NFR-04.1 | Preferred Fly.io region: GRU |
| NFR-05.1 | Single backend instance for campaign consistency |
